# Техническое задание: Stocker

Система автоматизированного отбора и загрузки фотографий на микростоки.

---

## 1. Цель

Превратить неразобранный поток семейных и съёмочных фотографий в поток отобранных, описанных и загруженных на микростоки снимков — с минимальным участием пользователя. Задача-философия: небольшой пассивный доход лучше нулевого; спешки нет, объёмы небольшие.

Проблема, которую решаем: у пользователя (жена — профессиональный фотограф с существующими стоковыми аккаунтами) в архивах лежит много потенциально продаваемых кадров, до которых «не доходят руки». Ручной разбор, отбор лучших из серий, написание описаний и загрузка — рутина, из-за которой всё простаивает.

---

## 2. Пользовательский сценарий (главный ориентир)

Именно так система должна выглядеть со стороны пользователя. Всё остальное в этом ТЗ подчинено этому сценарию.

1. Пользователь скидывает в одну папку-«входящие» **всё подряд** из разных источников, в ежедневном режиме: снятые кадры вперемешку с документами, фото счётчиков, фото мусора для жалоб на соседей, бытовыми семейными снимками. Никакой предварительной сортировки — свалил и забыл.
2. Система в фоне разбирает кучу: отсеивает всё нестоковое, схлопывает серии почти-одинаковых кадров до лучшего, и формирует **короткий список кандидатов на сток**.
3. Пользователь просматривает этот список и **одобряет** подходящие (может отклонить или переопределить выбор лучшего кадра в серии).
4. Одобренные снимки автоматически получают заголовки, описания, ключевые слова и **загружаются на настроенные стоки**.

Ручной труд пользователя сведён к двум действиям: **скинуть в папку** и **пролистать шортлист, апрувнуть**.

### Принципиальные оговорки к сценарию
- **Шаг «одобрить» обязателен** и остаётся всегда (автозаливка без подтверждения — только как опция на будущее, для самых уверенных случаев).
- **«Отсеяно» ≠ «удалено».** Забракованные снимки уходят в «нестоковую» корзину, а не стираются. Пользователь может её просматривать (важно на этапе калибровки, чтобы не терять ошибочно отсеянное годное). Оригиналы не удаляются никогда.
- Классификация — **поддержка решения, а не истина в последней инстанции.** Финальное слово за пользователем.

---

## 3. Архитектура

**Модульный конвейер поверх общей базы данных.** Не «настоящие» сетевые микросервисы (при десятках фото в неделю это лишняя инфраструктура), а набор независимо запускаемых модулей-этапов, которые обмениваются данными через одну общую БД и файловое хранилище превью. Каждый модуль можно запускать, дорабатывать и чинить отдельно.

**Локальный, инкрементальный, приватный.** Всё крутится на машине пользователя. Обработка идёт по мере поступления файлов, порциями (например, по 100 в день) — без требования «разобрать всё быстро». К облачным ИИ обращается только смысловая классификация и генерация метаданных; отбор лучших из серии — полностью локальный.

Поток данных:

```
Папка-входящие
     │  (M1: приём)
     ▼
  [assets]  ──(M2: классификация сток/не-сток, vision)──►  сток-кандидаты / корзина
     │
     ▼  (M3: группировка серий + отбор лучших, локальный CV)
  шортлист по группам
     │
     ▼  (M4: веб-ревью — одобрение пользователем)
  одобренные
     │
     ▼  (M5: генерация метаданных, vision)
  описанные + вшитые IPTC/XMP
     │
     ▼  (M6: загрузка на стоки)
  опубликованные
```

---

## 4. Технологический стек

- **Язык:** Python.
- **База данных:** SQLite (файл на диске, ноль настройки, достаточно для одного пользователя и таких объёмов).
- **Интерфейс:** локальное веб-приложение (открывается в браузере на машине пользователя). Обязательно умеет показывать снимки **группами** (серия + предложенный лучший + альтернативы), а не плоским списком.
- **ИИ-зрение:** Anthropic API, модель **Haiku 4.5** (достаточна для классификации и метаданных, дёшева), с использованием **Batch API** для массовой обработки архива.
  - ВАЖНО: используется платный API-ключ с оплатой по токенам. Подписка Claude Pro ($20/мес) для программного доступа **не подходит**.
- **Локальный анализ изображений:** классический CV для качества (резкость, экспозиция), перцептивные хеши / эмбеддинги для сходства, детекция лиц и их атрибутов (глаза/улыбка) для оценки кадра.
- **Работа с форматами:** JPEG, RAW (CR2/NEF/ARW и др. — через `rawpy`/встроенные превью), HEIC с iPhone (через `pillow-heif`). Для веб-интерфейса из любого исходника генерируются нормализованные JPEG-превью.
- **Метаданные:** запись IPTC/XMP в файлы через `exiftool`.
- **Платформа:** Windows 11.

## 5. Модули

### M1. Приём (intake)
**Назначение:** надёжно принять файлы из папки-входящих и завести их в базу.
**Вход:** папка-входящие с произвольными файлами.
**Выход:** записи в таблице `assets` со статусом «новый» + сгенерированные превью.
**Логика:**
1. Сканирует папку, находит новые файлы поддерживаемых форматов (JPEG/RAW/HEIC).
2. Считает хеш содержимого; точные дубли (тот же файл повторно) не заводятся дважды.
3. Читает EXIF: дата/время съёмки (нужна для группировки серий), камера/модель, ориентация, параметры съёмки.
4. Генерирует нормализованное JPEG-превью фиксированного размера для галереи (из RAW — через встроенное превью, без полного декодирования).
5. Записывает всё в базу.
**Не делает:** никаких решений о «стоковости» — только регистрация.

### M2. Классификация «сток / не-сток» (смысловой фильтр)
**Назначение:** отделить коммерчески пригодные кадры от бытовых семейных снимков и мусора (документы, счётчики, скриншоты, фото для жалоб).
**Вход:** снимки со статусом «новый».
**Выход:** для каждого снимка — вердикт и метаданные вердикта; статус меняется на «сток-кандидат» или «не-сток» (в корзину).
**Логика:** один запрос к Haiku-зрению на снимок возвращает структурированный результат:
- `stock_worthy` (да/нет);
- `reason` (короткое объяснение — зачем/почему);
- `has_logo` / `has_brand` / `has_text` (флаги комплаенса — совмещаем комплаенс-скан с классификацией в одном вызове);
- `category` (примерная стоковая категория);
- заметки о технических проблемах, если видны.
**Особенности:**
- Массовая обработка архива — через Batch API (вдвое дешевле).
- Порог/промпт калибруются на реальных снимках под критерии пользователя.
- Это **основной источник затрат** (вызов почти на каждое входящее фото).
- Результат — поддержка решения; ошибки в обе стороны возможны, поэтому «не-сток» уходит в просматриваемую корзину, а не в удаление.

### M3. Группировка серий и отбор лучших (механический фильтр)
**Назначение:** из серий почти-одинаковых кадров оставить лучший, сохранив при этом осмысленно разные варианты.
**Вход:** снимки-кандидаты (прошедшие M2).
**Выход:** снимки размечены по группам; в каждом наборе почти-одинаковых отмечен «лучший».
**Трёхуровневая модель:**
1. **Сцена (серия):** группировка кадров одной съёмочной ситуации по близости времени в EXIF + визуальному сходству.
2. **Набор почти-одинаковых:** внутри сцены — подгруппы кадров, различающихся лишь незначительно (та же поза/выражение).
3. **Лучший в наборе:** выбор одного кадра по оценке качества.
**Ключевое различение:** восемь однотипных «улыбается влево, чуть разный фокус» → схлопываются в один лучший; но «улыбается влево» и «смеётся вперёд» → это **разные наборы**, оба выживают как отдельные киперы. Это не дубли на выброс, а разные продаваемые варианты.
**Оценка качества (локально, без ИИ):** резкость/фокус, экспозиция, шум, открытые глаза, наличие/качество улыбки. Здесь используется **детекция лица и его атрибутов** — это НЕ опознавание личности (личности и релизы отложены, см. §7).
**Особенности:**
- Порог «избыточно похоже» против «уже другой кадр» — настраиваемый; стартуем консервативно, чтобы не выкинуть ценный вариант.
- Выбор лучшего — предложение; пользователь может переопределить в интерфейсе.

### M4. Веб-интерфейс ревью
**Назначение:** дать пользователю быстро просмотреть шортлист и принять решения.
**Функции:**
- Галерея кандидатов, сгруппированная по сериям: предложенный лучший кадр + альтернативы набора рядом.
- По каждому снимку виден вердикт классификатора и его причина, флаги (логотипы и т.п.).
- Действия: **одобрить**, **отклонить**, **переопределить выбор лучшего** в серии.
- Доступ к «нестоковой» корзине для выборочной проверки (важно на калибровке).
**Выход:** снимки со статусом «одобрен».

### M5. Генерация метаданных
**Назначение:** снабдить одобренные снимки данными, которые требуют стоки.
**Вход:** одобренные снимки.
**Выход:** заголовок, описание, 30–50 ключевых слов, категория — вписанные в файл как IPTC/XMP (через `exiftool`).
**Логика:** запрос к Haiku-зрению на снимок. Стоки читают вшитые метаданные при приёме.

### M6. Загрузка на стоки
**Назначение:** отправить отобранное и описанное на площадки.
**Вход:** снимки с готовыми метаданными.
**Выход:** снимки со статусом «загружен» + отметка площадки/даты.
**Логика:** по FTP+CSV или API конкретного стока (зависит от площадки). Доступы к стокам настраиваются один раз. Отслеживание статуса загрузки и обработка отказов модерации.
**Особенность:** конкретные площадки и способ загрузки уточняются на этапе реализации этого модуля.

## 6. Модель данных

База — SQLite. Центральная сущность — снимок; остальные наполняются по мере прохождения конвейера.

**`assets` (снимок):**
- путь к оригиналу, путь к превью;
- хеш содержимого (уникальный — дедупликация);
- тип файла (jpeg/raw/heic), размер, разрешение;
- дата съёмки (EXIF), камера/модель;
- статус в конвейере: `новый` → `классифицирован` → `сгруппирован` → `одобрен` → `загружен` (плюс ветка `не-сток`/корзина);
- результат классификации: `stock_worthy`, `reason`, флаги логотипов/брендов/текста, `category`;
- группировка: id сцены/серии, id набора почти-одинаковых, признак «лучший в наборе»;
- метаданные: заголовок, описание, ключевые слова, категория;
- загрузка: площадка, статус, дата.

Поля под результаты последующих модулей заводятся заранее пустыми — их заполняют соответствующие этапы.

*(Точная нормализация — например, вынос групп, лиц, загрузок в отдельные таблицы — определяется при реализации; здесь зафиксирован состав хранимой информации.)*

---

## 7. Вне текущего охвата (отложено)

- **Распознавание личностей и сопоставление с релизами.** Пользователь сам знает, на кого есть релизы; программа этого пока не касается. Когда вернёмся — опознавание строится **только на локальных эмбеддингах** (Claude/облако не идентифицируют частных лиц), и это тоже поддержка решения, не юридическая гарантия. (Детекция лиц для оценки кадра в M3 — это другое и остаётся.)
- **Автоматический забор фото из облаков** (Google Photos, iCloud). Пока — ручная папка-входящие. Отдельный модуль на будущее; из-за ограничений API реалистично работать с локальными экспортами.
- **Автоодобрение** без участия пользователя (возможная опция на будущее для самых уверенных случаев).
- **Обучение на решениях пользователя** (подстройка классификатора под «одобрил/отклонил») — улучшение на потом.

---

## 8. Нефункциональные требования и ограничения

- **Локальность и приватность:** данные и обработка на машине пользователя; в облако уходят только изображения, отправляемые в vision-API для классификации/метаданных.
- **Сохранность оригиналов:** оригиналы никогда не удаляются; отсев — это смена статуса, а не удаление.
- **Инкрементальность:** система работает порциями, устойчива к тому, что архив разбирается медленно и долго.
- **Стоимость (см. §9):** должна оставаться низкой; выбор Haiku + Batch API продиктован этим.
- **Платформа:** Windows 11 (учитывать при выборе библиотек — HEIC/RAW/детекция лиц бывают капризны при установке).
- **Характер объёмов:** ежедневно немного (единицы–десятки фото), плюс разовый большой разбор существующего архива без спешки.

---

## 9. Стоимость

Локальная часть (приём, группировка, отбор лучших) — бесплатна.
Облачная часть — vision-вызовы Haiku на классификацию (почти каждое входящее фото) и на метаданные (только одобренные):
- ориентировочно **~$0.005 за фото**, вдвое дешевле через Batch API;
- разовый разбор архива, например, на 20 000 снимков — порядка **$50–100** (батчем ближе к нижней границе);
- текущий режим (пара десятков фото в неделю) — центы.
Точная цифра уточняется после того, как станет известен объём архива. Классификация — основной и масштабируемый с числом фото источник затрат.

## 10. Порядок реализации

Строим по модулям, начиная с проверки главной гипотезы.

1. **Валидационная выборка (первый практический шаг).** Собрать пару сотен репрезентативных фото (явно стоковые, явно семейные, мусор), прогнать через приём и классификатор, сверить вердикты с мнением пользователя. Цель: убедиться, что ИИ адекватно отделяет сток от несток; измерить реальную стоимость; подстроить промпт под критерии пользователя — до вложений в разбор всего архива.
2. **M1 Приём + база + превью.**
3. **M2 Классификация «сток/не-сток».**
4. **M4 Минимальное веб-ревью** (просмотр кандидатов, одобрение) — чтобы замкнуть цикл «скинул → увидел шортлист → одобрил».
5. **M3 Группировка серий + отбор лучших** (и доработка ревью под показ по группам).
6. **M5 Генерация метаданных.**
7. **M6 Загрузка на стоки.**

MVP — пункты 2–4: полностью локальный + один vision-этап, уже даёт полезный результат «куча → шортлист кандидатов».

## 11. Открытые вопросы

- Объём существующего архива (влияет на оценку стоимости и стратегию превью).
- Конкретные стоки-площадки и способ загрузки (FTP+CSV или API) + доступы — для модуля M6.
- Проверка беспроблемной установки на Windows библиотек для HEIC, RAW и детекции лиц.
- Критерии «стоковости» под калибровку промпта классификатора (уточняются на валидационной выборке вместе с фотографом).
