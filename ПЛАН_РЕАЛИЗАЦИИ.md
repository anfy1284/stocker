# План реализации: Stocker

Разбивка проекта (см. `ТЗ.md`) на шаги. **Каждый шаг — один запрос к ИИ-агенту:** законченная единица работы, которую агент реализует и сам проверяет за один заход. Шаги идут по порядку; следующий начинаем, только когда предыдущий проверен. Внутри шага указано, что он делает, от чего зависит и по какому признаку считается готовым.

Фазы: **A** — фундамент, **B** — MVP «куча → шортлист → апрув» (на JPEG), **C** — полный отбор (RAW/HEIC, серии, лучший кадр), **D** — публикация, **E** — оптимизации и будущее.

---

## Фаза A. Фундамент

### Шаг 1. Каркас проекта и конфигурация
**Делает:** структура Python-проекта, управление зависимостями (venv/requirements), конфигурация (API-ключ Anthropic из переменной окружения, пути к папке-входящих, хранилищу превью и файлу БД), логирование, создание пустой SQLite-БД при первом запуске.
**Зависит от:** —
**Готово, когда:** проект ставится из чистого окружения, запускается, читает конфиг, создаёт пустой файл БД; секреты не в коде.

## Фаза B. MVP «куча → шортлист → апрув» (только JPEG)

Цель фазы — замкнуть главный сценарий на самом простом формате и проверить, что ИИ вменяемо отделяет сток от несток, до вложений в остальное.

### Шаг 2. Приём JPEG
**Делает:** схема таблицы `assets`; сканирование папки-входящих; хеш содержимого и дедупликация; чтение EXIF (дата съёмки, камера, ориентация); генерация простого уменьшенного JPEG-превью; запись записей со статусом «новый».
**Зависит от:** Шаг 1.
**Готово, когда:** при сбросе JPEG в папку появляются записи в БД с EXIF и превью; повторный тот же файл не заводится дважды.

### Шаг 3. Классификатор «сток / не-сток» (синхронный)
**Делает:** для каждого снимка со статусом «новый» — один запрос к Haiku-зрению, возвращающий структурированный вердикт (`stock_worthy`, `reason`, флаги логотип/бренд/текст, `category`); запись результата в БД; смена статуса на «сток-кандидат» или «не-сток» (корзина). Обработка по одному (без батча).
**Зависит от:** Шаг 2.
**Готово, когда:** на тестовой пачке из явно стоковых, семейных и мусорных фото вердикты записываются и осмысленно раскладывают снимки.

### Шаг 4. Прогон валидационной выборки и калибровка промпта
**Делает:** прогон пары сотен репрезентативных реальных фото через приём и классификатор; сравнение вердиктов с мнением фотографа; подстройка промпта под её критерии «стоковости»; фиксация реальной стоимости.
**Зависит от:** Шаг 3.
**Готово, когда:** классификатор согласуется с оценкой фотографа на приемлемом уровне (контрольная точка go/no-go для всего проекта); промпт зафиксирован.

### Шаг 5. Минимальный веб-интерфейс ревью
**Делает:** локальное веб-приложение: галерея снимков-кандидатов с превью, вердиктом и причиной; кнопки «одобрить» / «отклонить»; отдельный просмотр «нестоковой» корзины. Пока плоский список (группировка — в фазе C).
**Зависит от:** Шаг 3 (данные для показа).
**Готово, когда:** пользователь открывает страницу, видит кандидатов, одобряет/отклоняет, статусы в БД меняются; корзина просматривается.

> После Шага 5 MVP замкнут: «скинул кучу JPEG → получил шортлист → апрувнул».

## Фаза C. Полный отбор

### Шаг 6. Приём RAW и HEIC + нормализованные превью
**Делает:** расширение приёма на RAW (CR2/NEF/ARW и др.) и HEIC (iPhone); чтение их EXIF; генерация нормализованных JPEG-превью для веб-интерфейса (из RAW — через встроенное превью).
**Зависит от:** Шаг 2.
**Готово, когда:** RAW и HEIC принимаются наравне с JPEG, для них есть превью, отображаются в интерфейсе.

### Шаг 7. Группировка серий
**Делает:** объединение кадров одной съёмочной ситуации в «сцены» (близость времени в EXIF + визуальное сходство); внутри сцены — выделение «наборов почти-одинаковых»; запись id сцены и id набора в БД.
**Зависит от:** Шаг 6 (или Шаг 2, если пока на JPEG).
**Готово, когда:** на реальной серии кадры корректно группируются; осмысленно разные кадры («улыбается влево» vs «смеётся вперёд») попадают в разные наборы, а почти-одинаковые — в один.

### Шаг 8. Оценка качества и выбор лучшего в наборе
**Делает:** локальная оценка каждого кадра (резкость/фокус, экспозиция, шум, открытые глаза, улыбка — через детекцию лица и его атрибутов, без опознавания личности); пометка «лучший в наборе».
**Зависит от:** Шаг 7.
**Готово, когда:** в каждом наборе почти-одинаковых отмечен один лучший кадр; выбор выглядит разумным на тестовых сериях.

### Шаг 9. Веб-ревью по группам
**Делает:** доработка интерфейса: показ по сериям — предложенный лучший кадр плюс альтернативы набора рядом; возможность переопределить выбор лучшего.
**Зависит от:** Шаги 5, 8.
**Готово, когда:** пользователь просматривает шортлист сгруппированным, видит и меняет выбор лучшего, одобряет.

## Фаза D. Публикация

### Шаг 10. Генерация метаданных
**Делает:** для одобренных снимков — запрос к Haiku-зрению (заголовок, описание, 30–50 ключевых слов, категория); вшивание в файл как IPTC/XMP через `exiftool`.
**Зависит от:** Шаг 5 (наличие одобренных).
**Готово, когда:** у одобренного снимка в файле присутствуют корректные IPTC/XMP-метаданные.

### Шаг 11. Загрузка на сток
**Делает:** отправка отобранного и описанного на конкретную площадку (FTP+CSV или API); настройка доступов; отслеживание статуса и обработка отказов модерации.
**Зависит от:** Шаг 10; уточнение площадок и доступов (открытый вопрос из ТЗ).
**Готово, когда:** одобренный снимок с метаданными успешно уходит на выбранный сток, статус фиксируется.

## Фаза E. Оптимизации и будущее

### Шаг 12. Batch-классификация
**Делает:** перевод массового разбора архива на Anthropic Batch API (создание батча, опрос, сбор результатов) для вдвое меньшей стоимости.
**Зависит от:** Шаг 3.
**Готово, когда:** большой набор классифицируется через батч, результаты корректно раскладываются в БД.

### Вне текущего охвата (из ТЗ, на потом)
- Автоматический забор фото из облаков (Google Photos / iCloud).
- Распознавание личностей и сопоставление с релизами (только локальные эмбеддинги, поддержка решения).
- Автоодобрение для самых уверенных случаев.
- Обучение классификатора на решениях пользователя «одобрил/отклонил».
